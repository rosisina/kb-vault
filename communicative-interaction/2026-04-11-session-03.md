### 2.4 Aurora v2 integration strategy (Option C — knowledge-base coreization)

James presented the strategic question: given that the prior `Projects/vault/` prototype evaluation showed Karpathy LLM-wiki beating Aurora's GraphRAG on quality, accuracy, indexing cost, and GraphRAG-limit coverage, how should the two projects (`knowledge-base` and `defense-kg-platform`) be integrated? Three named options:

- **A**: integrate Aurora's Angular+Neo4j into knowledge-base
- **B**: integrate knowledge-base into Aurora
- **C**: port Aurora's good parts (Angular UI, Neo4j viz) into knowledge-base, creating "Aurora v2"

**Decision: Option C, with specific architectural pattern — wiki as source of truth, graph as materialized view.**

Key reasoning:
- Option B is rejected because James's own prototype evaluation already showed graph-first retrieval is worse for this domain (7-layer + Record No. 13,495 + 3-axis truth structure). Forcing wiki content into the graph paradigm discards the narrative atomicity that made Karpathy work.
- Option A is the right direction but underspecified — it implies "import code as-is" which is unrealistic if Aurora's Angular is tightly Cypher-coupled.
- Option C is the correct framing: **role inversion**. Aurora keeps the brand and the visualization/UI capabilities; the wiki becomes upstream. The graph becomes a downstream cache, rebuildable from wiki via `atoms-to-cypher.py` (already designed in CLAUDE.md §self-compiling).

**Architecture**: 7-layer model — raw → wiki → compiler → Neo4j → retrieval (Karpathy primary, GraphRAG secondary for relationship queries) → single API → 3-pane UI (chat | evidence | graph+timeline).

**Key invariants**:
1. Single-direction data flow (raw → ... → UI; no reverse).
2. Graph is cache (delete-and-rebuild from wiki anytime).
3. Triple pseudonym defense gateway (PreToolUse hook + API filter + UI redaction).
4. Every API response includes `(commit_sha, schema_version, atom_hash)` for legal-proceeding reproducibility.

**Single-pane UI**: 3-pane layout matching James's claude.ai/gemini.google.com reference, with 6 modes (Ask / Trace / Contradict / Timeline / Layer / Falsification) and a "book export" bonus mode for LaTeX/PDF generation with auto-footnoted Record No. citations.

Saved as standalone proposal: `output/aurora-v2-integration-proposal-2026-04-11.md` (19 KB) + `.docx` (51 KB).

### 2.5 Sequencing — "knowledge-base v1 first then verify, then Aurora v2"

James asked whether the right approach is purely serial: complete knowledge-base, verify it, then start Aurora v2.

**Decision: partial parallelism, not pure serialization.**

Agreement points:
- Wiki content authorship cannot be put on hold while infrastructure is built. James's instinct here is correct — empty grandeur is the opposite failure mode of vault/ speed.
- Verification must precede infrastructure layering. An unverified vault under a compiler/API/UI is "a building on a wrong foundation".
- UI must wait until atom volume is sufficient (~100 atoms) for retrieval to feel magical.

Disagreement points:
- **Compiler must run as a pilot at atom 30**, not at atom 100 or after "v1 complete". Reason: debugging the wiki↔Neo4j round-trip is *easier with smaller volume* (visual 1:1 trace possible). Postponing means retroactive atom-format fixes across 30–50 atoms when the compiler is finally written.
- **Phase 2 API serves dual purpose**: it is *both* part of Aurora v2 *and* a writing accelerator for knowledge-base. Postponing it makes knowledge-base authorship slower because James is left with manual grep.

**Recommended sequence (atom milestones, parallel tracks):**

| Atom milestone | knowledge-base track | Aurora v2 track |
|---|---|---|
| now (~few) | `/compile raw/02–07` proceeding | none (infra ready) |
| 30 | continue compile | Phase 1 starts: `atoms-to-cypher.py`, `build-record-index.py`, Aurora 6-agent triage |
| 50 | continue compile + 1st `/reverify` pass required | Phase 1 ends; Phase 2 starts: FastAPI single endpoint, pseudonym gateway |
| 80 | continue compile | Phase 2 ends; API used as authoring accelerator |
| 100 | 2nd `/reverify` pass — knowledge-base v1 declared complete | Phase 3 starts: 3-pane UI, book export mode |
| 150 | wiki stabilization | Phase 3 ends; Aurora v2 complete |

**knowledge-base v1 completion criteria proposed:**
1. raw/02–07 fully ingested
2. ≥10 atoms per layer (≈70 atoms)
3. ≥5 entries in `_contradictions.md`
4. `/reverify` passes (all verdicts CONSISTENT or NEW_ATOM_NEEDED with the latter resolved)
5. `/lint` passes (zero pseudonym violations, zero layer mismatches)
6. all core book narrative decomposed into atoms
