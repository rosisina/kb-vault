## 2. Major decisions made during this session

### 2.1 Strict pseudonym rule is absolute — even in "가명 매핑 문서화" contexts

When Track D wrote `wiki/claims/lee-jiyoung-kim-sujin-single-point-of-control.md` with real names deliberately placed inside the Aurora `claimDesc` field and one `## Supporting evidence` bullet as "가명 매핑 문서화" (for mapping clarity), I redacted in-place to `mapping id N` references. The rationale: the strict rule in CLAUDE.md §Conventions says "Real names never appear in this vault, ever, anywhere" with no exception for documentation intent. Pseudonym mapping clarity can be preserved via mapping-id references that a reader can resolve against `../defense-kg-platform/kg/pseudonym_mapping.json`. This precedent is now codified as "no exceptions" for any future atom that attempts similar inline documentation.

### 2.2 Log.md append-only rule is overridden by pseudonym rule for surgical redaction

The A.6 Re-verify pass found 4 real names (mapping ids 43, 44, 64, 65) in `wiki/log.md` lines 71 and 83 — both inside pseudonym-mapping-metadata entries. This created a direct rule conflict: log.md is append-only, but real names are strictly forbidden. Resolution: **pseudonym rule wins on narrow in-place redaction** (replace real names with mapping-id references), while the append-only invariant is preserved on entry structure (no entries deleted, no line order changed). A new append-only `fix` entry documented the redaction itself. This establishes the precedent that strict-enforcement rules (pseudonym) override structural conventions (append-only) on narrow token-level edits when the two conflict.

### 2.3 Filename stability over semantic renaming

When Branch 1's P6 blind re-measurement expanded `2436ho-cluster-six-anchors.md` from 6 anchors to 8, the filename became semantically stale. Decision: **keep the filename** for URL stability (~15 inbound wikilinks) and document the six→eight expansion trajectory in an explicit "Filename note" section at the top of the atom body. Rationale: renaming would require coordinated updates across 15+ inbound references and risk breaking any out-of-session reader's mental URL cache, whereas in-body annotation preserves URL invariants while making the evolution visible. Luhmann's "names are traces of the archive's history" principle cited as supporting rationale.

### 2.4 Stage-don't-mutate contract for all `scripts/` output

Decision made when designing ⑦ `atoms-to-cypher.py`: the script must never write to Neo4j directly, and must never modify existing `wiki/` files. Instead, it stages output to `output/wiki-promotions-YYYY-MM-DD.cypher` and leaves promotion to James via manual `cypher-shell -f`. Extended to ⑥ `build-record-index.py` (wiki/_record-index.md is the one exception, explicitly auto-generated and documented as such in the validator exclusions) and ⑧ `rebuild-hubs.py` (three output/ staging artifacts). Rationale: respects the "hard-to-reverse operations" principle (Neo4j writes + wiki hub overwrites are both hard-to-reverse) and guarantees non-collision with James's parallel work (other Claude sessions editing the same wiki files do not conflict because the scripts only read).

### 2.5 Book is authoritative on propositional matters; filename metadata is not primary evidence

The L7 rebuttal date adjudication established this: James's own filename convention `(20220929)` on the raw/05 rebuttal document was **metadata, not primary source**. Reading the 3,811-line document interior confirmed no printed date anywhere — the document does not self-date. Under the Re-verify authority principle, the book's §3.7.1.1 inline date `2022.9.25` governs. Broader implication: any apparent discrepancy between wiki and book should trigger a check of whether the wiki's source is the *book's interpretation* of a primary source or the primary source itself; if it's the book's interpretation, book authority applies without exception.

### 2.6 Check-pseudonym.sh SIGPIPE 141 bug explained the raw/02 ingest real-name leak

When the lint CRITICAL finding surfaced 7 real names in 2 files (`jang-woo-jin.md` + `lee-ji-young.md`) — and earlier the A.6 Re-verify found 8 real names in 6 files + 2 real names in 1 more file via Path-C pilot — the root cause was traced to a SIGPIPE 141 exit-code bug in the PreToolUse hook. The hook's printf|tr|python3 pipeline, when python detected a real name and exited 2, caused SIGPIPE to propagate back to printf, and the shell pipefail combined with the termination chain produced exit 141 instead of exit 2. Claude Code does not recognize exit 141 as a permission-deny, so the write proceeded silently. **Fix**: rewrite the hook as a single python block that reads the PreToolUse JSON envelope from an environment variable rather than stdin. Tested with real-name input (exit 2 + stderr) and pseudonym-only input (exit 0). This root-cause fix explains why the three separate pseudonym leak batches during raw/02 first ingest all went undetected at write time — they were all written during the broken-hook window.
