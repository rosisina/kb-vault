### 2.4 Obsidian property/tags integration as CP-3 prerequisite (Phase 0)

James identified that Obsidian's property and tags features (YAML frontmatter, nested tags, Dataview queries) could strengthen atom interconnection for both Obsidian and UI simultaneously. Investigation revealed:
- **0/297** atoms had YAML frontmatter (all metadata trapped in Cypher blocks)
- **Dataview plugin** already installed but unused
- YAML frontmatter creates a **single source of truth** serving 3 systems: Obsidian (Dataview/Graph/search), UI (Python→JSON→Angular), Neo4j (Cypher block preserved)

### 2.5 YAML schema design — 4 additional fields discovered

Beyond the initially proposed schema, analysis of 297 atoms revealed 4 additional extractable fields:
- `secondary-layers` (32 atoms with cross-layer presence)
- `source-type` (book/recording/regulation/investigation/sop/kakao)
- `event-date` (1,231 date mentions across atoms)
- `has-verbatim` (28 atoms with direct witness quotes)

### 2.6 Language policy: structured bilingual, not full translation

Korean/English mixed atom titles evaluated. Decision: add `title-ko`/`title-en` and `lang` property rather than standardizing all atoms to one language. Reasoning: evidence preservation (Korean originals), international accessibility (English Key Takeaways), and UI i18n switching.

### 2.7 Aliases: 3-type system for Obsidian autocomplete

All 1,763 wikilinks used pipe syntax (`[[slug|표시명]]`) — Obsidian autocomplete couldn't find atoms by Korean name. Solution: 3 aliases per atom (resultId, Korean short, English short) enabling `[[한글 검색]]` directly.
