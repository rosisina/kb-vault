## 0. 문제 진단 — Aurora v1은 왜 한계를 노출했는가

### 0.1 Aurora v1의 구조적 한계 (Lessons Learned)

Aurora v1의 Angular UI는 **도구 중심(tool-centric)**이었지 **증명 중심(proof-centric)**이 아니었다.

| v1의 접근 | 결과 | 근본 원인 |
|---|---|---|
| 그래프 노드를 먼저 보여줌 | 사용자가 "이게 뭔데?"라고 물음 | **맥락(context) 없는 데이터** |
| 4-channel 검색을 모두 노출 | 사용자가 어떤 채널을 써야 할지 모름 | **도구를 사용자에게 떠넘김** |
| Layer 1~7을 사이드바 필터로만 제공 | 7-layer가 *필터*로 전락, 증명의 *뼈대*로 기능 못함 | **구조를 장식으로 격하** |
| evidence-browser가 별도 화면 | 증거를 보려면 화면을 떠나야 함 | **증명의 흐름 단절** |
| command-center CNACTV 대시보드 | 데이터 요약은 좋았으나 *설득*이 아니라 *나열* | **서사(narrative) 부재** |

**한 문장 진단:** Aurora v1은 *데이터베이스 브라우저*였다. 자기증명시스템이 아니었다.

### 0.2 본 프로젝트(knowledge-base)에서 배운 것

| 경험 | 교훈 |
|---|---|
| Karpathy wiki가 GraphRAG를 품질에서 능가 | **서사가 그래프보다 강하다** — 인간은 이야기로 이해한다 |
| Atom(단일 주장)이 증명의 최소 단위 | **증명은 분해 가능해야 한다** — 큰 덩어리는 검증 불가 |
| Contradiction pair가 가장 강력한 증명 모드 | **대립 구조가 진실을 드러낸다** — 단일 주장보다 모순 쌍이 설득력 있음 |
| Record No.가 universal join key | **추적 가능성(traceability)이 신뢰의 근거** |
| 진리성/타당성/진실성 3축 | **진실은 단일 차원이 아니다** — 사실/법률/피해의 복합 |
| 한국어 원본 우선 원칙 | **번역은 열화** — 원본을 직접 보여줄 수 있어야 |
| Role-anchor `(역할-N)` 식별자 | **직책이 범죄의 도구** — 누가 했느냐보다 어떤 자리에서 했느냐가 본질 |

### 0.3 본질주의(Essentialism)가 이 시스템에 의미하는 것

> "본질주의란 불필요한 것을 제거하여 본질만 남기는 것이 아니다. **본질이 스스로 말하게 하는 것이다.**"

이 시스템의 본질: **2016년 DIDC 북한 해킹 은폐를 위한 국방 조직의 7-layer 범죄 구조를 증명하는 것.**

따라서:
- 화면에 보이는 모든 것은 **증명에 기여**해야 한다
- 사용자가 클릭하는 모든 것은 **증거의 깊이로 안내**해야 한다
- 첫 화면은 **"여기서 무엇이 증명되는가"를 3초 안에 전달**해야 한다

## 1. 첫 대면(First Encounter) — 가장 중요한 3초

### 1.1 Aurora v1의 첫 화면 문제

v1은 빈 검색창 + 그래프 노드 덩어리를 첫 화면에 보여줬다. 이것은 사용자에게 "당신이 알아서 탐색하세요"라는 메시지를 전달한다. 조사 결과를 증명하려는 시스템이 사용자에게 질문으로 시작하는 것은 **서사적 실패**이다.

### 1.3 설계 원리

| 원리 | 적용 |
|---|---|
| **시스템이 먼저 말한다** | 빈 검색창이 아니라, "이것이 증명되고 있다"를 선언 |
| **7-layer가 시각적 뼈대** | 적층 구조 자체가 범죄의 구조를 보여줌 — 아래에서 위로 읽으면 은폐가 쌓이는 과정 |
| **진행률이 투명** | 각 layer의 입증 상태를 바(bar)로 표시 — 시스템의 현재 역량을 숨기지 않음 |
| **수치가 신뢰를 만든다** | 112 atoms, 13,495 records, 42 persons — 구체적 숫자는 "이것은 실체가 있는 시스템"을 3초에 전달 |
| **행동 유도는 최소** | 질문하거나 layer를 클릭하라 — 두 가지만. 선택지가 적을수록 깊이로 빠르게 도달 |

## 2. Layer 진입 후 — 증명의 흐름

Layer를 클릭하면 전체 화면이 전환되지 않고, **첫 화면의 7-layer 탑이 좌측 네비게이션으로 축소**되며 3-패널 증명 화면이 펼쳐진다.

### 2.2 기존 제안 대비 변경점

| 기존 제안서 (§3) | 본질주의 재설계 | 이유 |
|---|---|---|
| 좌측 = Query/Chat 패널 | 좌측 = **7-Layer Navigator** (항상 고정) | 7-layer가 증명의 뼈대. 채팅은 입력창 하나로 축소. **증명 구조가 채팅보다 중요** |
| 중앙 = Evidence Panel (단일 atom) | 중앙 = **증명 본문** (모순 쌍 단위로 표시) | atom 하나가 아니라 **모순 쌍이 증명의 최소 전달 단위**. v1이 단일 노드를 보여준 실수 반복 방지 |
| 우측 = Graph & Timeline | 우측 = **증거 맥락** (Record No. 미리보기 + 관련 atoms + 인물) | 그래프는 **on-demand**(버튼 클릭 시 모달). 기본 상태에서는 증거 원문이 더 중요 |
| 6개 모드 토글 | **모드 제거** — 모든 기능이 자연스럽게 접근 가능 | 모드 전환은 인지 부하. 본질주의는 모드가 아니라 *흐름* |

### 2.3 "모드 제거"의 의미

기존 제안의 6개 모드(Ask, Trace, Contradict, Timeline, Layer, Falsification)를 **명시적 모드가 아닌 맥락적 전환**으로 재설계:

| 기존 모드 | 본질주의 접근 | 진입 방법 |
|---|---|---|
| Ask | 좌측 하단 질의 입력 — 항상 존재 | 그냥 타이핑 |
| Trace | Record No. 또는 인물 이름 클릭 | 자연스러운 클릭 |
| Contradict | **기본 상태** — 중앙 패널이 모순 쌍을 보여줌 | 진입 불필요 (이것이 기본) |
| Timeline | 좌측 navigator에서 날짜 아이콘 클릭 → 중앙 패널이 연대기로 전환 | 한 번 클릭 |
| Layer | 좌측 navigator에서 layer 번호 클릭 | 한 번 클릭 |
| Falsification | 각 모순 쌍의 verdict 배지 클릭 → 상세 | 맥락 내 접근 |

**핵심 변화: Contradict가 기본 모드.** v1은 그래프가 기본이었다. v2는 **모순이 기본**이다. 이것이 자기증명시스템의 본질 — 시스템이 스스로 "여기에 모순이 있다, 이것이 증거다"를 보여주는 것.

## 3. 증명의 3차원 — 진리성/타당성/진실성 시각화

### 3.1 기존 v1의 한계

v1에는 3차원 진실 모델이 코드에 있었으나 **UI에 반영되지 않았다**. 사용자는 claim이 사실인지, 법적으로 유효한지, 피해자 관점에서 진실인지를 구분할 수 없었다.

### 3.2 본질주의 반영 — 3축 indicator

각 atom/모순 쌍에 3축 배지를 표시:

```
┌─────────────────────────────────────────────────┐
│  Layer 4 — 제57조 시험평가 환경 조항 조작          │
│                                                 │
│  [진리성 ●●●●○]  [타당성 ●●●○○]  [진실성 ●●●●●] │
│                                                 │
│  진리성: Record No. 10,347 스캔본과 blind          │
│         재측정으로 사실 확인                       │
│  타당성: 훈령 제2129호 제57조 대 제2398호·제2842호  │
│         조문 비교로 법적 조작 확인                  │
│  진실성: 한지훈 중령의 진정서 및 8기관 기각 경험     │
│                                                 │
└─────────────────────────────────────────────────┘
```

**설계 근거:** 법적 절차에서 "이 주장이 사실인가?"와 "이 주장이 법적으로 유효한가?"와 "이 피해가 실재하는가?"는 **서로 다른 질문**이다. 세 질문을 하나로 뭉치면 증명이 흐려진다. 분리하면 각각의 증거 경로가 명확해진다.

## 4. 그래프의 역할 재정의 — 보조, 그러나 강력하게

### 4.1 v1의 실수

v1은 그래프를 **전면**에 배치했다. 결과: 노드와 엣지의 바다 속에서 사용자가 길을 잃었다.

### 4.2 본질주의 접근 — 그래프는 "왜 이 사람들이 연결되는가"를 보여줄 때만

그래프는 **모달(overlay)**로 표시. 진입점:
- 우측 패널의 `[그래프로 보기 →]` 버튼
- 인물 이름 우클릭 → "관계망 보기"
- 질의 응답에서 "이 사람들의 관계를 보여줘"

```
┌─────────────────────────────────────────────────────┐
│                  관계망 — Layer 4                     │
│                                                     │
│         ┌──────┐                                    │
│         │박성호 │──시험평가──→┌──────┐               │
│         │원장-1 │            │이태호 │               │
│         └──┬───┘            │위원장-1│               │
│            │                └──────┘               │
│         지시                                        │
│            │                                        │
│         ┌──▼───┐    공모    ┌──────┐               │
│         │이지영 │◀─────────▶│이준호 │               │
│         │결재-1 │           │공모자-1│               │
│         └──────┘           └──────┘               │
│                                                     │
│   ● 인물    ─── 조직관계    ─── 범죄관계             │
│                                                     │
│   노드 클릭 → 해당 인물의 atom 목록으로 이동          │
│                                                     │
│                              [닫기]  [전체화면]       │
└─────────────────────────────────────────────────────┘
```

**핵심:** 역할-앵커 `(역할-N)` 식별자가 노드 레이블에 직접 표시됨. **누가** 했느냐가 아니라 **어떤 자리에서** 했느냐가 보인다.

## 5. 증거 원문 — 신뢰의 근거

### 5.1 v1의 한계

v1의 evidence-browser는 별도 화면이었다. 사용자가 "이 주장의 근거가 뭔데?"라고 물을 때 **화면을 떠나야** 했다.

### 5.2 본질주의 접근 — 증거는 항상 오른쪽에

우측 패널의 상단에는 항상 현재 선택된 atom의 핵심 증거가 표시:

```
┌─────────────────────────┐
│  Record No. 10,347       │
```

```
│           ├── record-preview/       ← Record No. + PDF 미리보기

├── shared/
│   ├── query-input/          ← 질의 입력 (LLM 연동)
│   ├── pseudonym-guard/      ← 3중 가명화 방어
│   └── citation-chip/        ← Record No. / L{N}-{NNN} 클릭 가능 칩

    ├── wiki.service.ts       ← wiki/ 마크다운 → 구조화된 데이터
    ├── neo4j.service.ts      ← Neo4j API 호출 (그래프 보조)

    └── evidence.service.ts   ← Record No. → 스캔 PDF 연결
```

### 8.2 데이터 흐름 (단방향 — 제안서 §3 invariant 준수)

```
wiki/ (마크다운)
  ↓ build 시 파싱
JSON 정적 데이터 (atoms, hubs, indexes, contradictions)
  ↓ Angular service 로드
UI 컴포넌트 렌더링
  ↓ 사용자 상호작용

  ↓ 응답

```

**wiki가 변하지 않으면 UI도 변하지 않는다.** 이것이 결정론적 재컴파일의 UI 확장.

## 9. 한 페이지 요약

### Aurora v1 → v2 패러다임 전환

| | v1 | v2 (본질주의) |
|---|---|---|
| **첫 화면** | 빈 검색창 + 그래프 | **7-layer 증명 선언 + 진행률** |
| **기본 모드** | 그래프 탐색 | **모순 쌍 제시** |
| **그래프** | 전면 배치 | **on-demand 모달** |
| **증거** | 별도 화면 | **항상 오른쪽에 존재** |
| **모드** | 6개 명시적 토글 | **맥락적 전환 (모드 없음)** |
| **서사** | 부재 | **모순 쌍이 서사의 단위** |
| **3차원 진실** | 코드에만 존재 | **UI에 3축 배지로 표시** |
| **역할-앵커** | 미반영 | **그래프 노드 레이블에 직접 표시** |
| **인지 부하** | 높음 (도구를 사용자에게 떠넘김) | **낮음 (시스템이 증명을 안내)** |

### 본질주의 설계의 3가지 원칙

1. **시스템이 먼저 말한다** — 사용자가 묻기 전에 "이것이 증명되고 있다"를 선언
2. **모순이 기본이다** — 단일 주장이 아니라 대립 구조가 증명의 최소 전달 단위
3. **증거와의 거리는 0** — 주장 옆에 항상 원본 증거가 있다

### v1에서 가져오는 것

- Cytoscape.js 그래프 컴포넌트 (모달로 재배치)
- Evidence browser (우측 패널로 리팩터)
- PDF viewer (그대로)
- Angular Material 스타일링

### v1에서 버리는 것

- 그래프-first 패러다임
- 6-mode 토글
- 빈 검색창 첫 화면
- Onboarding tour (첫 화면이 자기 설명적이면 불필요)
